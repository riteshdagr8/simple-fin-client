SimpleFinClient

Personal Finance Manager — Architecture Review & System Flow

1. Architecture Overview

Frontend React 19

  • Vite dev server (port 6173) + build tool
  • Hash routing — no server-side route config needed
  • AuthContext — JWT stored in localStorage
  • 2 themes — minimal / colorful via CSS custom properties
  • Zero UI libraries — pure React components

Backend Express 5

  • REST API — 8 route groups, all JWT-protected
  • SQLite — WAL mode, foreign keys, 13 tables
  • AES-256-GCM — secrets encrypted at rest
  • Rate limiting — per-endpoint and per-connection
  • Cron scheduler — auto-sync + email summaries

External Services Integrations

  • SimpleFIN Bridge — bank transaction data (Open Banking)
  • OpenAI / Anthropic — AI transaction categorization
  • SMTP (Gmail) — verification, password reset, daily summaries

Key Design Decisions

  • Fire-and-forget sync — background, non-blocking
  • 3-tier categorization — rules → manual → LLM
  • Job tracking — LLM jobs polled every 2s
  • Token invalidation — password change kills old JWTs
  • Dedupe indexes — prevents duplicate accounts/transactions

8. Code Review

Critical Issues

🔴 Start scripts contain hardcoded secrets
start-express.ps1 has JWT_SECRET, ENCRYPTION_KEY, and SMTP credentials in plaintext. While the file is gitignored, the content is visible in the workspace. These should come from .env only.
🔴 Client/server password validation mismatch
Register.jsx requires 8 chars minimum, but server requires 10+ chars with at least one letter and one number. Users can pass client validation and get rejected by the server. Same issue in ResetPassword.jsx (client: 8 chars, server: 10+).
🔴 No CSRF protection — JWT in localStorage
JWT tokens stored in localStorage are vulnerable to XSS attacks. Consider HttpOnly cookies with SameSite=Strict for the auth token, or add CSRF token validation.

Warnings

🟡 Email verification is effectively disabled
Users are created with email_verified=1 at registration (line 72 of auth.js). The verification email is sent but the user is already fully authenticated. The VerifyEmail page exists but serves no real purpose.
🟡 Dead code in rules.js
buildPatternsForCategory() has two queries where the first (txns) references t.user_id which doesn't exist on the transactions table. The variable txns is never used — dead code that would throw if reached.
🟡 No token cleanup for expired email_tokens and reset_tokens
Expired tokens accumulate in the database forever. Add a periodic cleanup job or prune on startup.
🟡 /api/seed available in production
The seed endpoint creates demo data and is auth-protected, but available in production mode. Consider disabling when NODE_ENV=production.
🟡 Single SQLite database for all users
All users share one finapp.db. A compromised database exposes all users' financial data. The design doc describes a planned per-user DB architecture but it's not implemented.

Improvements

🔵 No test suite
Zero test files exist. Critical paths to test: crypto.js (encrypt/decrypt roundtrip), rules.js (pattern extraction), llm.js (response parsing), and route integration tests.
🔵 No health check endpoint
Add GET /api/health that returns { status: 'ok', db: 'connected' } for monitoring and container orchestration.
🔵 No graceful shutdown
No SIGTERM handler to close the database connection and stop cron jobs cleanly. Important for production deployments.
🔵 .env.example missing ENCRYPTION_KEY
The boot guard requires ENCRYPTION_KEY but it's not documented in .env.example. New users will get a startup error without knowing why.
🔵 Incomplete auth-service
The auth-service/ directory references files that don't exist (./db.js, ./keys.js). Either complete it or remove it to avoid confusion.
🔵 WebSocket for LLM job progress
Current polling (every 2s) is functional but wasteful. Consider Server-Sent Events or WebSocket for real-time job updates.

9. What's Done Well

Security

  • AES-256-GCM encryption for secrets at rest
  • Boot guard refuses default/missing secrets
  • Rate limiting on auth endpoints
  • CSP headers via Helmet
  • Explicit CORS allowlist
  • Token invalidation on password change

Architecture

  • Clean separation: routes / services / middleware
  • Singleton DB with WAL mode
  • Fire-and-forget sync (non-blocking UI)
  • Job tracking for background LLM work
  • 3-tier categorization (rules → manual → LLM)
  • Dedupe indexes prevent data corruption

Developer Experience

  • Concurrent dev server (Express + Vite)
  • Seed data endpoint for testing
  • Comprehensive .env.example
  • SQL migration pattern (ALTER TABLE on startup)
  • Clean React component architecture

Feature Completeness

  • Full auth flow (register, login, reset, verify)
  • SimpleFIN integration with rate limiting
  • AI categorization with multiple providers
  • Rules engine for auto-categorization
  • Email summaries with configurable content
  • CSV import for manual transactions
  • Multi-theme support
